## The two bugs (confirmed against source at HEAD a4bb9db)

`pkg/worker/worker.go` `processJob` success branch (~409-444):
`cancelHeartbeat()` runs at ~399 (BEFORE the completion writes). Then `SaveJobResult` (~411),
`completeWithRetry` (~416), and `handleSubJobCompletion(ctx, job, completeErr==nil)` (~441) are
THREE separate writes. The fan-out counter increment (`IncrementFanOutCompleted/Failed`, raw
non-idempotent `count = count + 1` in `handleSubJobCompletion`) is a SEPARATE step from the
ownership-guarded state transition.

- **BUG 1 (over-count / mis-account):** on a TRANSIENT (non-ErrJobNotOwned) Complete error, the row
  stays running but the code still calls `handleSubJobCompletion(..., succeeded=false)` â†’increments
  FAILED for a job that SUCCEEDED. Reaper later reclaimsâ†’re-runâ†’increments COMPLETED. Net accounted
  = total+1 (INV-FANOUT-COUNTS violation) + interim wrong fail-fast.
- **BUG 2 (reaper-vs-Complete dup-run + wedge):** `cancelHeartbeat()` stops lock renewal before the
  writes; if the lock is near expiry the reaper reclaims between handler-return and Complete â†’ another
  worker re-runs (dup side effect); original Complete returns ErrJobNotOwned and drops its result;
  for a sub-job whose SaveJobResult committed but Complete lost ownership, the early-return at ~424
  skips accounting â†’ ~2m GetStalledFanOutParents latency.

ROOT CAUSE: non-idempotent fan-out counter increment performed as a separate write from the
ownership-guarded state transition, with the heartbeat cancelled before the completion writes.

## The fix (4 coordinated parts; no `core.Storage` interface change)

**(1) Storage â€” add concrete `*GormStorage` methods (type-asserted by the worker, legacy fallback):**
- `CompleteWithResult(ctx, jobID, workerID string, result []byte) (*core.FanOut, error)`
- `FailTerminalWithResult(ctx, jobID, workerID, errMsg string) (*core.FanOut, error)`

Each = ONE `withSerializationRetry` + `Transaction`:
1. ownership-guarded job UPDATE â€” same guard as `Complete`/`Fail` terminal arm:
   `WHERE id=? AND locked_by=? AND status='running'`. `CompleteWithResult` sets
   status=completed, completed_at, locked_by='', locked_until=nil, and `result` ONLY if `result!=nil`
   (preserve prior SaveJobResult semantics). `FailTerminalWithResult` mirrors `Fail`'s terminal arm:
   last_error=`security.SanitizeErrorMessage(errMsg)`, status=failed, completed_at, clear lock.
2. `RowsAffected==0` â†’ return `core.ErrJobNotOwned`, increment NOTHING.
3. else read the job's `fan_out_id`; if nil/"" return (nil, nil).
4. **LIVENESS-GUARDED** counter UPDATE in the SAME tx (Attack-3 fix):
   `UPDATE fan_outs SET completed_count = completed_count + 1, updated_at=? WHERE id=? AND status='pending'`
   (failed_count for the Fail variant). No-op if the fan-out already terminalized.
5. re-read + return `*core.FanOut` (nil if no fan-out).
`ErrJobNotOwned` is not a serialization failure â†’ passes through `withSerializationRetry` unwrapped.

**(2) Worker success path** â€” replace SaveJobResult+completeWithRetry+handleSubJobCompletion with a
single `CompleteWithResult` (via a `retryWithBackoff` wrapper). **MOVE `cancelHeartbeat()` to AFTER it
returns.** ErrJobNotOwnedâ†’log+return. TRANSIENT errâ†’log + **prompt `Release(jobID,workerID)`** (P2's
ownership-guarded runningâ†’pending) so it re-dequeues in one poll cycle, then return (Attack-2 fix).
Successâ†’CallCompleteHooks + Emit(JobCompleted) + `checkFanOutCompletion(returnedFanOut)`.

**(3) Worker terminal-fail paths** â€” route BOTH `handleError`'s terminal arm (retryAt==nil) AND the
no-handler branch through `FailTerminalWithResult`; same cancelHeartbeat-after + prompt-Release-on-
transient. **Retryable failures (retryAt!=nil) keep `Fail` (statusâ†’pending) and MUST NOT increment.**

**(4) `handleSubJobCompletion`** â†’ reduced to resume-only (fetch fan-out + checkFanOutCompletion),
called only on the legacy-fallback (mock/non-Gorm) path; no longer the increment site on the atomic path.

## Idempotency (exactly-once-into-a-live-fanout), all interleavings

normal=1; transient-Complete-failâ†’rollback(0)+Release+rerun=1; reaper-before-txâ†’0 here, rerun=1;

## PREREQUISITE #2 RESOLVED (verified on-machine at HEAD a4bb9db)
The liveness guard (`WHERE fan_outs.status='pending'`) is **CORRECT**. Verified:
- `cmd/chaostest/main.go:352` `checkFanOutCounts` asserts **STRICT** `completed_count + failed_count
  + cancelled_count == total_count` (HARD invariant).
- `cmd/chaostest/main.go:164` chaos.fanout uses `FailFast() + CancelOnParentFailure()`, 5 subs, sub
  #4 fails deterministically.
- `pkg/core/fanout.go:45` `TerminalStatus`: FailFast (default arm) returns `FanOutFailed` the instant
  `FailedCount>0` â€” i.e. terminalizes with `accounted < total`. Strict equality is then reached
  **via cancellation**: `CancelOnParentFailure` â†’ `CancelSubJobs` increments `cancelled_count` for the
  remaining in-flight subs until accounted==total. (That's why chaos is green today.)
- Therefore: once a fan-out is `failed`, a straggler sub-job that completes afterward MUST NOT
  increment `completed_count` â€” its slot was already counted as `cancelled`. The liveness guard drops
  exactly that increment, keeping `accounted==total`. WITHOUT the guard, the straggler would push
  `accounted` to `total+1` and violate the strict HARD invariant. **Guard confirmed necessary and
  sufficient; no escalation to COUNT(*)-GROUP-BY needed.**
- Note: `collect_all` only terminalizes at `accounted==total`, so it has no straggler window; the
  guard is a no-op there (correct). `threshold` can terminalize early like fail_fast; same reasoning
  applies â€” a post-terminal straggler's slot is cancelled-accounted, guard drops the dup.
